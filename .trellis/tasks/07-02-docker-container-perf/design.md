# Docker 容器列表切换性能优化设计

## Goal

把 Docker 工具面板的容器视图改成对大列表友好的渲染模式，降低右侧工具切换时的 React commit、DOM 节点数量和 Radix 交互包装成本。

## Architecture

Docker 面板保持现有数据获取和操作接口不变，主要改两层：

1. 容器列表改为窗口化渲染。容器卡片高度固定，列表容器保持滚动容器语义，通过可见窗口计算只渲染当前可视区域附近的一小段条目。顶部保留统计和刷新，底部保留刷新入口和必要的空状态。
2. 面板生命周期进一步降成本。Docker 面板在非 active 状态下不应继续参与重列表渲染；远程文件面板需要保留路径和展开状态，但 SSH tab 切换时只渲染当前 active 面板，非 active tab 只保留状态缓存，不保留组件实例、文件树、ContextMenu 和工具内容。
3. Docker 数据刷新从会话 tab 激活路径里解耦。已有缓存时只恢复缓存快照，不因缓存过期立即刷新；没有缓存时延迟加载，后台自动刷新只作用于当前可见 Docker 子页。

交互上保留现有行为：详情、日志、终端、启动/停止/重启、删除都继续存在。Tooltip 和右键菜单保持项目共享组件，但只作用于当前窗口化后实际挂载的可见条目。

## Boundaries

- `src/features/tools/DockerToolPanel.tsx`
  - 负责容器列表窗口化渲染、列表状态、可视窗口计算、条目操作入口。
- `src/features/layout/WorkspaceShell.tsx`
  - 负责 Docker 面板的挂载策略，确保非 active 时不背着重列表持续参与渲染。
- `src/features/files/RemoteFilePanel.tsx`
  - 如果需要，保留面板显示/隐藏策略的轻量联动，但不把 Docker 性能逻辑放到这里。
- `src/styles/app.css`
  - 负责窗口化列表的容器样式、占位块、滚动容器、固定卡片间距。
- `src/shared/ui/Tooltip.tsx` / `src/shared/ui/TabContextMenu.tsx`
  - 这次不改协议，只减少它们在大列表里的实例数量。

## Data Flow

1. Docker 列表从现有刷新流程拿到容器数组。
2. 容器数组进入窗口化列表组件，组件根据滚动位置、容器固定高度和缓冲区算出当前可渲染区间。
3. 只渲染区间内的容器条目，区间外用上下占位块保持滚动高度稳定。
4. 当 Docker 面板非 active 时，父层不再让重列表长期留在可见切换路径上；当 SSH tab 非 active 时，父层不渲染对应远程文件面板，路径、展开状态和目录缓存由 `stateKey` 状态缓存恢复。
5. 当 SSH 会话 tab 切回一个已有 Docker 缓存的连接时，先恢复缓存；容器刷新由用户手动刷新或当前可见子页的后续轮询负责。

## Trade-offs

- 窗口化会比“先加载 N 条再追加”更彻底，能解决 200+ 条的长列表问题，但实现复杂度略高。
- 直接卸载非 active Docker 子树最省性能，但会丢一些本地状态；因此优先保留轻量壳，把状态切换与重列表分离。
- Tooltip/右键菜单共享化收益次于窗口化，但能进一步降低大列表上的 portal 成本。

## Verification

- 用电脑使用工具在至少两台已有会话上测试：一个容器较少、一个容器很多。
- 实测切换右侧工具和 Docker 视图，确认大列表会话不再出现十秒级卡顿。
- 本地运行 `npm run check`。
- 视实现结果决定是否补充更细的性能日志或专门的回归测试。
