# 优化 Docker 容器列表切换性能

## Goal

让 Docker 工具面板在容器数量很大时仍然保持切换流畅，避免右侧工具在文件/工具/Docker 之间切换时出现明显卡顿。

## Requirements

- Docker 容器列表需要支持大列表场景，不再一次性渲染所有容器卡片。
- Docker 面板在非激活状态下不能继续背着完整重列表参与高成本渲染。
- 容器列表切换到工具/文件/其他右侧面板时，应避免因 Docker 卡片、Tooltip、右键菜单的全量挂载导致明显停顿。
- SSH 会话 tab 切换时，如果 Docker 面板已有该连接的缓存数据，不能因为缓存过期而立即重新执行远端 `docker ps`。
- SSH 会话 tab 切换时，远程文件面板需要保留路径和展开状态，但右侧区域只渲染当前 active 会话的文件面板，不能背着多个隐藏实例导致切换变慢。
- 优化后需要保留现有功能：查看详情、终端操作、日志、启动/停止/重启、删除、刷新。
- 优化方案必须可在本机通过电脑使用工具实际验证。

## Acceptance Criteria

- [ ] 276 个容器规模下，切到右侧工具和从工具切回 Docker 的体感卡顿明显下降。
- [ ] Docker 容器列表不再一次性全量挂载所有条目，滚动过程中只维持可见范围附近的条目渲染。
- [ ] 容器每行仍保留原有操作能力和视觉风格。
- [ ] 在当前仓库的电脑使用工具实测中，切换 Docker 工具面板不再出现十秒级卡顿。
- [ ] 在已有 Docker 缓存的会话间切换 tab 时，前端只恢复缓存，不在切换路径上立即触发容器刷新。
- [ ] 右侧远程文件面板只渲染当前 active SSH tab，非 active tab 的路径和展开状态通过缓存恢复。

## Notes

- 这是前端性能优化任务，重点在 Docker 容器列表和右侧面板生命周期。
- `prd.md` 只记录需求和验收，不写实现细节。
- 需要在实现前完成设计文档和实施计划。
