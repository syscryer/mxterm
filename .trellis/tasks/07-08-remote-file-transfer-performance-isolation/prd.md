# 远程文件传输性能隔离

## Goal

远程文件上传、下载、排队、进度刷新和传输完成后的历史状态不能拖慢主界面。大文件传输期间，终端、文件树、标签切换、右侧面板折叠展开等主界面操作应保持响应；传输完成后也不能因为残留的高频状态、定时器、监听器或过多历史项导致界面变慢。

用户价值：文件传输是后台工作，不能让用户感觉整个桌面工具被传输任务“占住”。传输面板可以更新，但更新范围必须被限制在传输功能自身。

## Confirmed Facts

- 当前远程传输状态、队列、运行集合、任务 map、进度事件合并和传输面板都集中在 `WorkspaceShell.tsx`。
- 后端 SFTP 进度事件已经有 200ms 或 1MB 或完成时才 emit 的节流，卡顿更可能来自前端高频状态更新导致的大组件重渲染。
- 已有短期优化把进度事件合并为 120ms 批量刷新，但状态仍属于 `WorkspaceShell`，不能从根上隔离渲染。
- 用户反馈包括两类体验问题：
  - 传输过程中主界面有卡顿。
  - 传输过一次大文件后，即使传完了，界面响应也不如刚启动时快。
- 外部成熟产品调研结论只作为思路：常见做法是独立传输状态容器、独立传输组件、后端/事件快照和 UI 进度显示分层。不得复制外部 GPL 代码。

## Requirements

- 传输状态必须从 `WorkspaceShell` 的 React state 中移出，改为传输功能独立拥有的外部 store 或等价隔离机制。
- 高频进度事件只能触发传输面板相关订阅者更新，不能触发 `WorkspaceShell` 主组件重渲染。
- 队列、并发调度、运行中任务集合、任务取消、任务重试、清理完成项等传输控制逻辑应从 `WorkspaceShell` 中抽离到独立模块。
- 传输面板应保持现有视觉风格和交互，不引入新的 UI 体系。
- 传输完成后必须清理运行中任务、队列引用、进度合并缓存、脉冲定时器等可能残留的资源。
- 完成/跳过/取消/失败历史项需要有上限或清理策略，防止一次或多次大批量传输后长期拖慢界面。
- 现有功能语义必须保持：
  - 上传成功刷新远端父目录。
  - 下载成功保留本地路径并支持复制、打开、定位。
  - 取消排队任务只改本地状态；取消运行任务调用 `remoteFileCancelTransfer`。
  - 失败任务可按原 retry 信息重试。
  - 上传进度到达真实 100% 后，在命令确认前仍显示“等待远端确认”并避免提前成功。
- 不改变后端传输协议、不改变并发设置含义。
- 允许引入小型、宽松协议、对性能隔离有直接收益的前端状态管理依赖；依赖必须服务于局部订阅和减少重渲染，不能增加无关 UI 框架。

## Acceptance Criteria

- [ ] `WorkspaceShell` 不再持有 `remoteFileTransfers` 这类高频传输列表 state。
- [ ] `WorkspaceShell` 不再订阅 `remote_file:transfer_progress` 并直接 setState；进度事件由传输模块内部监听、合并和应用。
- [ ] 传输进度更新时，主 shell 的终端、文件树、标签栏等不因传输列表变化而重渲染。
- [ ] 传输结束、失败、取消后，运行任务和队列引用会被移除；不留下会继续触发 UI 更新的 interval/timeout。
- [ ] 完成项清理和自动裁剪能限制长期历史数量，传完大文件后主界面不会因为传输历史无限增长而变慢。
- [ ] 文件传输面板仍显示方向、类型、大小、速度、百分比、状态、错误、复制/打开/定位/重试/取消/移除等现有能力。
- [ ] `npm run check` 通过。
- [ ] `node scripts/check-startup-module-boundary-source.mjs` 通过。
- [ ] `node scripts/check-remote-file-editor-source.mjs` 通过。

## Notes

- 这是复杂任务，需要 `design.md` 和 `implement.md`。
- 本任务主要是前端架构/性能隔离，优先不动 Rust 后端。
- 用户已允许为了性能引入新包；当前推荐使用 Zustand 的 vanilla store + selector 订阅能力。
- 如果验证发现后端事件本身仍有压力，再单独评估后端 snapshot/速度字段等增强。
